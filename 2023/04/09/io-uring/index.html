    【2022秋】异步io: io_uring

    异步时代

    阻塞/非阻塞

    当我们读linux文件,(specifically socket).我们使用glibc 的::read,或者linux系统调用read.

    一般来说,linux系统调用(sys_read)后如果内核缓冲区(sk_buff)没有数据,linux会调用schedule函数切换到其它进程,因此当前进程也就不能运行,对外表现为阻塞.

    但是我们可以设置内核调用套接字为非阻塞的(在创建套接字时指定).

    那么当我们调用read后linux内核发现内核缓冲区没有数据,直接返回EAGAIN错误

    同步/异步.完成端口(iocp)

    异步

    以asio框架为例,看看异步框架怎么读取socket:

    异步io基本流程 : 你想要读某个socket –> 注册一个读的函数 –> 读完socket后 调用回调函数

    完成端口(windows)

    完成端口是一种真正意义上的异步模型。在重叠IO模型中，当Windows socket应用程序在调用WSARecv函数后立即返回，线程继续运行。另一线程在在完成端口等待操作结果，当系统接收数据完成后，会向完成端口发送通知，然后应用程序对数据进行处理。

    以epoll为例,看看同步读取socket:

    为什么epoll不能说是完全的异步

    epoll 只是表明socket可读,但是并没有完成读的这一操作.还需要一个同步读.

    linux异步

    缺点:

    没有文件异步,虽然linux有(page cache,page prefetch,mmap等优化手段,但磁盘读取速度还是低于主存读写好几个量级)

    这意味着在高性能服务器中,文件读写还是会成为拖慢整个系统的瓶颈.在传统的半同步(或者异步)网络库中,使用线程池来绕过(workaround)这个问题.比如(nodejs的底层网络库libuv,asio,rust 运行时tokio):

    但是无论怎样做,都不可避免地增加了系统的复杂度.linux始终缺少统一的异步方式.直到io_uring的到来.

    io_uring

    history: from https://lwn.net/Articles/810414/

    基本思想

    io_uring 的基本逻辑与 linux-aio 是类似的：提供两个接口，一个将 I/O 请求提交到内核，一个从内核接收完成事件。

    每个 io_uring 实例都有两个环形队列（ring），在内核和应用程序之间共享：

    • 提交队列：submission queue (SQ)
    • 完成队列：completion queue (CQ)

    应用提交系统调用(可能一次提交多个),然后可以立即返回,内核在后台执行系统调用.

    好处

    1. io_uring最大的好处是统一了各种异步调用.一套api完成基本所有的事情.不用利用thread pool 来完成
    2. 批处理系统调用如 read ,write ,send ,recv ,accept .openat ,stat
    3. 内核空间和用户空间共享,提交收割系统调用不用进入内核
    4. …

    示例

    echo-server:

    可以看到io_uring 的web 服务器和epoll的架构大致相同,不过io_uring的阻塞读是在背后完成的,而epoll模型要等待同步read读完.

    性能

    随机io读,

    完全绕过page cache.此时基本所有读都会直接落到磁盘上

    顺序io读,

    完全命中page cache 基本所有读都会在page cache中读到

    用了 io_uring 之后，用户就无需担心同步和异步各占多少比例了，因为它在任何场景下都表现良好。
